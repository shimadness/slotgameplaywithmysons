# 5リールモード TENGU KING 化 — 設計書（実装前）

> TRIPLE SLOT 内「② 5リール」モードを、コナミ**フィーチャープレミアム TENGU KING**型の
> ビデオスロットへ寄せるための設計。**この文書は設計のみ。コードはまだ変更しない。**
> 確定方針（2026-06-27 ユーザー決定）:
> 1. 配当評価 ＝ **左詰め全リール方式**（10本固定ライン廃止）。**2個そろいから配当**
> 2. テーマ ＝ **和・天狗**
> 3. **ベット ＝ 3×3 DROP と同じく「1スピン固定額」**
> 4. **フリーゲーム中の一律倍率は無し**（ビッグウィンの源はワイルド花火だけ）
> 5. **ワイルド ＝ リール2〜4に出現。①通常の代用 ②各ワイルドが花火で×2/×3を持ち、
>    盤面に出た全ワイルドの倍率を掛け合わせて「総配当」に乗算**（例: 2・3・4列に×2/×3/×2＝総配当×12）（§2.4）
> 6. 演出アセット（花火/突入ムービー）は**別制作・並行**。コードは差し込み口だけ先に作る（§6.5）

## 1. 参照元（TENGU KING とは）

- 5リール ビデオスロット（コナミアミューズメント「フィーチャープレミアム」シリーズ）。
- **配当ルールがシンプル**：「**一番左のリールから同じシンボルが揃った数**」で配当が決まる。
- **天狗シンボルが3個以上**でフリーゲーム突入 → メダル大量獲得。
- コンセプト＝**シングルメダル機の分かりやすさ**＋ビデオスロットの派手な演出。
- 出典: コナミ公式トピック / 稼働開始リリース（2024-02-08）/ gamebiz。

## 2. 評価方式の刷新（核心）

### 2.1 現状（廃止する）
`paylines.ts` の `PAYLINES`（10本固定ライン）を1本ずつ走査し、左から連続一致を判定。

### 2.2 新方式：左詰め全リール（“出た数”方式）
ラインの概念を捨て、**各シンボルが左リールから連続して何リールに出現したか**で配当を決める。

判定アルゴリズム（疑似コード）:

```
baseWin = 0
for each symbol S が「リール1（最左）」のどこかのセルに出現する場合:
    count = 1
    for reel = 2..5:
        if そのリールの3セルのどこかに S または WILD が出現:
            count++ ; 該当セルを記録
        else:
            break              # 連続が途切れたら終了
    if count >= 2:             # ★2個そろいから配当が出る
        baseWin += pay[S][count] × bet     # ライン数・ways数は乗じない（シンプル配当）

# ▼ ワイルド花火の倍率（§2.4）。盤面に出ている全ワイルドの倍率を掛け合わせる
wildMult = 1
for each WILD が盤面に出現:
    wildMult *= そのワイルドの花火倍率(2 or 3)
totalWin = baseWin × wildMult            # ★総配当に乗算
```

- **WILD は代用**。リール1には WILD を出さない（§2.4）ので、リール1は必ず実シンボルが本体。
- **配当は2個そろいから**（`count >= 2`）。`pay[2]` を新設（§3）。
- **同一シンボルは1回のみ計上**（リール1に同シンボルが2セルあっても二重取りしない）。
- **「ライン数」「ways数（組合せ数）」は乗じない**。シングルメダル機の素朴さを優先し、
  `pay[count] × bet` の単純配当にする（`bet` = 1スピン固定額）。
- 天狗シンボル（旧スキャッター）は本判定の対象外（§4で別処理）。

### 2.3 「揃った数」の体感が変わる点
- 旧: 特定ラインに乗らないと当たらない → 当たりが分かりにくい。
- 新: **左から並べばどの行でもOK** → 「左2つ光れば当たり」が直感的。TENGU KING の肝。

### 2.4 ワイルド花火の倍率（TENGU KING 名物演出）★確定
ワイルドは **2つの役割** を持つ：

1. **通常のワイルドカード**：左詰め判定で実シンボルの代用になり、並び（count）を伸ばす。
2. **花火の倍率**：盤面に出た各ワイルドが ×2 または ×3 を持ち、**盤面の全ワイルドの倍率を
   掛け合わせた値を「総配当（baseWin）」に乗算**する。

ポイント（実機準拠）:
- **ワイルドは「天狗フリーゲーム中だけ」出る★確定（ユーザー指示 2026-06-27）**。
  通常ゲームにはワイルドを一切出さない＝**花火×Nは通常では発生しない**。
  実装：通常帯 `wildWeight=0`（ワイルド除外）／フリー帯 `wildWeight=12`（engine.ts）。
- **出現位置**：ワイルドは **リール2・3・4** に出現（リール1/5には出さない）。1スピンに**複数同時に出うる**。
- **当たりに参加していなくても倍率は乗る**：例「1〜2列目で2個そろい、ワイルドは4列目（並びと無関係）」
  でも花火は出て、その倍率が**総配当に掛かる**。＝倍率は“ライン単位”ではなく“スピン全体”に作用。
- **掛け算で累積**：例「2列＝×2 / 3列＝×3 / 4列＝×2 → 総配当 ×（2×3×2）＝×12」。
- **倍率の決め方**：各ワイルドの倍率は演出前に**ロジックで先に抽選**（例: ×2=70% / ×3=30%、RTPで調整）。
  抽選結果を演出へ渡す（`onWildMultiplier(mults: number[], cells)`）。**演出は結果を表示するだけ**。
- **花火は当たりがある時だけ**★確定：**baseWin>0 のときのみ**花火を出す。
  ワイルドが出ても baseWin=0 なら花火・倍率演出は無し（空振り演出は出さない）。
- フリー専用の一律倍率は設けない（§4.2）。フリー中の唯一の伸びがこのワイルド花火。

## 4. 天狗フリーゲーム（フィーチャー）

### 4.1 突入 ★確定
- **天狗シンボル（👺）が画面内に3個以上**（位置不問）で突入。
- **付与スピン数**：3個=**8** / 4個=**15** / 5個=**25**（`freeSpinsFor` をこの値に書き換え）。
- **突入率 ≈ 1/141（天狗 weight 5）★更新（2026-06-27 ユーザー指示で 1/1024→約1/128 に引き上げ）**。
  天狗の出現重みが調整つまみ。帯枚数が離散なので weight 5=1/141 / weight 6=1/47 と飛ぶ → 5を採用（「1/128くらい」）。
  - 突入を頻繁化したぶん、フリー1回の爆発力は抑える（§4.2 `FREE_WILD_WEIGHT` を下げる）＝RTP維持。
  - 突入が軽い演出で流れないよう、突入ムービー（§6.5）でしっかり“ご褒美感”を出す。

### 4.2 フリーゲーム中の挙動 ＝ **一律倍率は無し★確定（P2実装済み）**
- **フリー専用の一律倍率は設けない**（ユーザー決定）。ビッグウィンの源は **§2.4 のワイルド花火倍率だけ**。
- フリー中は **ワイルド多めの「フリー帯」に切替**（`engine.setFreeMode(true)` ＋ `board.setStrips()`）。
  ワイルド重み **`FREE_WILD_WEIGHT=12`**（フリー帯 各リール5枚／約0.70個/スピン＝1.4スピンに1個）。
  終了時に通常帯へ戻す（`finishRush`）。
  > 設計（2026-06-27 ユーザー）：突入1/141(≈5〜15分に1回)は据え置き、**フリーを「ランキング一撃エンジン」**に。
  > ワイルド高めで単発最大≈9331×（5リール最大ベット200→約186万＝**100000の壁を超えてランキング入り可**）。
  > ワイルドを増やすほどフリー配当が跳ねRTPが上がる（=12でRTP≈138%）。つまみは `FREE_WILD_WEIGHT`。
- ベット消費なしの無料スピン。フラット倍率は撤去済（`startRush(fs, 1)`）。
- 突入演出（ムービー）は §6.5 のフックで差し込む（P3）。

> DROP の **セブンラッシュ基盤**（`state.startRush()` / FREE SPIN メーター / BET変更不可 /
> 自動collect）をそのまま流用。違いは「一律倍率を掛けない／ワイルド多めの専用帯」にする点。

### 4.3 リトリガー ＝ **あり★確定（P2実装済み）**
- フリー中に天狗3個以上で**追加スピン上乗せ**（`retriggerRush(fs)`、3=8/4=15/5=25）。バナー「👺 天狗 上乗せ +N」。
- フリー帯にも天狗を残しているのでリトリガー可能。

## 5. RTP（ペイアウト率）方針

- 計測ハーネス `rtp_5reel.mjs` を新設（多数セッション平均＋セッション間ブレ幅、シードで再現可）。
  使い方: `npx tsx rtp_5reel.mjs [spinsPerSession] [sessions] [payScale]`。
- フリーゲーム寄与・ワイルド倍率寄与・突入率・花火率・maxWin を分離集計。

### 5.1 計測結果（2026-06-27 最新）★実測
固定帯＋pay表§3＋**通常ワイルド無し／フリー帯weight12／天狗weight5（突入1/141）** で計測（30セッション×40万スピン）:

| 指標 | **現状** |
|------|------|
| **RTP（平均）** | **138.6%**（セッション134.9〜143.8%＝均一） |
| 当たり率 | 78.9% |
| **フリー突入率** | **0.71%（≒141スピンに1回 ≈ 1/128 ≈ 5〜15分に1回）** |
| フリー配当 寄与 | **38%（フリー＝蓄積＆ランキング一撃エンジン）** |
| フリー中ワイルド | 約0.70個/スピン（1.4スピンに1個・`FREE_WILD_WEIGHT=12`） |
| 単スピン最大 | **9331×ベット**（最大ベット200で約186万→ランキング100000壁超え可） |

> RTP138%は「フリー＝一撃エンジン」設計の帰結（普段はベース≈86%で緩く減り、フリーで取り返す）。
> ワイルドをさらに増やすと wild視認性↑だがRTPも↑（15→163% / 18→189%）。下げたい時は `FREE_WILD_WEIGHT`↓
> か フリースピン数↓ か 配当一律スケール↓ で調整（つまみは独立）。

> 経緯：①当初は通常にもワイルド→RTP104.8%。ユーザー指摘で**ワイルド＝フリー専用**化（通常0/フリー帯）。
> 通常からワイルドを抜くとRTPが60%へ落ちたため素の配当を約1.6倍にスケールして96.4%へ。
> ②ユーザー指示で**突入率を1/1024→約1/128(1/141)**に引き上げ。頻発で膨らんだRTPを `FREE_WILD_WEIGHT 12→5`
> で戻し、突入率1/141・RTP100.7%・フリー寄与14.7%に着地。検証：通常帯ワイルド0／フリー帯あり（決定的に確認）。

- 100.7%は家庭/会社用としては許容（やや甘め）。きっちり下げたい場合は配当一律スケールで微調整可（RTPは配当に線形）。

### 5.2 実装上の重要決定（P1で確定）
- **配当は小数倍率**（0.05刻み）。「2個そろい・全リール・任意行」で当たり率が高く、RTP96%域は
  整数倍率では表現不能なため。**RTPはベット非依存**（pay×bet と cost×bet で相殺）。
- **5リール単一ベット ＝ ベット段 × 20クレジット**（`FIVE_REEL_UNIT=20`）。
  小数pay(0.05刻み)×20＝**整数クレジット配当**。額面のみに作用しRTPは不変。
- **リール帯は固定シード生成**（`STRIP_SEED`）＝全プレイヤー共通。これをしないと
  セッション毎にRTPが1225〜5222%とブレた（実機スロット同様、帯は設計値にする）。
  帯は重み比例の正確枚数で構成＋決定的シャッフル（枚数ブレ排除）。出目のランダム性は停止位置で担保。

### 6.5 演出アセット戦略（動画 vs エンジン内）★確定方針
ユーザー指示：**アセット（花火・突入ムービー）は別制作・並行**。コードは**差し込み口だけ先に作る**。

- **ワイルド花火（×2/×3）→ エンジン内で実装（動画にしない）**
  - 理由：特定セル上に重ねる＋倍率がロジック決定。**ワイルドは複数同時に出て各々が花火を上げる**ので、
    セル位置に追従できるCanvas/Lottieが必須。動画だと座標合わせと組合せ管理が破綻する。
  - 実装：Canvas/CSS パーティクル or **Lottie**。フック `onWildMultiplier(mults: number[], cells)` で
    各ワイルドの花火を発火 → 倍率を順に見せて総配当が膨らむ演出（配当は §2.4 で確定済み）。
    まずプレースホルダ（簡易フラッシュ＋倍率テキスト）で通す。
- **フリーゲーム突入ムービー → プリレンダー動画 or Lottie（差し替え式）**
  - 実装：フック `onFreeGameEnter()` で全画面オーバーレイを再生 → 終了で本編へ。
  - 動画採用時の必須条件：`muted` + `playsinline`、**mp4(H.264)＋webm** 両提供、短尺（容量配慮、
    Capacitorネイティブの肥大化を避ける）、`file://` 再生確認。
  - 軽さ・差し替え・和テーマ調整を重視するなら **Lottie 推奨**。
  - コードは**プレースホルダ（バナー＋SE）で先に成立**させ、アセット完成後に差し替え。
- 設計：演出トリガーを `ui/effects.ts` 付近に**抽象フック層**として置き、ロジック↔アセットを疎結合に。
